# Go APM Server UI Fix Plan

## 목적

현재 Web UI의 구조적 문제를 우선순위 기반으로 정리하고, 실제 개선 작업을 순서대로 진행할 수 있는 실행 기준을 정의한다.

## 배경

현재 UI는 시각적으로는 다크 대시보드 방향을 갖고 있지만, 운영 화면으로 쓰기에는 다음 문제가 우선적으로 드러난다.

- `설정` 메뉴가 실제 라우트와 연결되지 않아 빈 화면이 발생한다.
- API 실패 시 에러 상태가 사용자에게 명확히 전달되지 않고, 일부 값은 `undefined`로 노출된다.
- 모바일 및 좁은 화면에서 사이드바와 본문 레이아웃이 붕괴한다.
- 트레이스, 로그, 상세 화면이 데이터 밀도에 비해 반응형 정보 구조를 거의 제공하지 않는다.
- 작은 타이포와 낮은 대비, 과한 장식 요소로 인해 운영 가독성이 떨어진다.
- 일부 애니메이션 클래스는 실제로 정의되어 있지 않아 유지보수 비용만 남긴다.

## 목표

- 빈 라우트와 네비게이션 불일치를 제거한다.
- 로딩, 빈 상태, 에러, 재시도, 마지막 정상 데이터 상태를 명확히 구분한다.
- 모바일과 데스크톱에서 모두 사용 가능한 레이아웃으로 재구성한다.
- 데이터 중심 화면의 정보 우선순위를 정리해 빠르게 스캔 가능한 UI로 개선한다.
- 시각 스타일을 정리해 가독성과 상태 인지를 높인다.

## 범위

이번 UI 개선 범위는 다음 화면과 공통 레이아웃을 포함한다.

- 앱 셸: 사이드바, 헤더, 메인 컨테이너
- 대시보드
- 트레이스 목록
- 트레이스 상세
- 로그 화면
- 공통 상태 UI: 로딩, 빈 상태, 에러, 재시도, 상태 배지

## 제외 범위

- 백엔드 API 스펙 변경
- 인증/권한 기능 추가
- 신규 메트릭 화면 대규모 확장
- 디자인 시스템 패키지 분리 또는 컴포넌트 라이브러리 전면 도입

## 우선순위

### 1. 네비게이션 및 라우트 정합성

가장 먼저 사용자가 클릭했을 때 깨지는 지점을 없앤다.

- `설정` 메뉴를 실제 라우트와 연결하거나, 준비 전까지 메뉴를 숨긴다.
- 존재하지 않는 경로에 대한 기본 처리 방식을 정한다.
- 헤더와 메뉴에서 현재 위치를 더 명확히 인지할 수 있게 한다.

### 2. 데이터 상태 처리 개선

운영 UI에서 가장 중요한 것은 실패를 실패로 보이게 만드는 것이다.

- API 요청 실패 시 에러 배너, 재시도 버튼, 마지막 정상 갱신 시간 표시를 추가한다.
- `undefined`, 빈 차트, 무의미한 `-` 값 노출을 제거한다.
- 대시보드, 트레이스, 로그 화면의 상태 모델을 통일한다.

### 3. 반응형 레이아웃 재설계

현재 고정 폭 사이드바와 고정 높이 화면은 모바일에서 거의 사용할 수 없다.

- 모바일에서는 사이드바를 드로어 또는 상단 토글 형태로 전환한다.
- `h-screen`, `calc(100vh - ...)` 기반 고정 높이를 최소화한다.
- 테이블 중심 화면은 작은 화면에서 카드형 또는 축약형 레이아웃을 제공한다.

### 4. 데이터 화면 정보 구조 개선

트레이스와 로그는 데이터가 많을수록 구조가 더 중요하다.

- 트레이스 목록은 핵심 컬럼 우선순위를 정의하고, 보조 정보는 접거나 상세로 이동시킨다.
- 로그 화면은 속성 표현 방식과 행 배치를 단순화한다.
- 트레이스 상세 워터폴은 작은 화면에서도 읽을 수 있는 레이아웃으로 분기한다.

### 5. 시각 시스템 정리

장식보다 운영성 중심으로 다듬는다.

- 타이포 스케일, 간격, 대비, 상태 색상을 재정의한다.
- 지나치게 작은 텍스트와 낮은 대비를 줄인다.
- 사용되지 않거나 정의되지 않은 애니메이션 클래스를 제거하거나 실제 정의한다.

## 단계별 산출물

### Phase A

- 라우트 정리
- 공통 에러/빈 상태 컴포넌트 초안
- 대시보드 데이터 실패 처리 정리

### Phase B

- 반응형 앱 셸
- 모바일 레이아웃 기준 정립
- 트레이스/로그 목록형 화면 1차 개편

### Phase C

- 트레이스 상세 워터폴 반응형 개선
- 시각 시스템 정리
- 애니메이션, 간격, 배지 표현 통일

## 완료 기준

- `/settings` 또는 대체 메뉴 정책이 확정되고, 빈 라우트가 없어야 한다.
- API 실패 시 모든 주요 화면에서 사용자에게 실패 사실과 재시도 수단이 보여야 한다.
- 390px 폭 기준에서 메인 콘텐츠가 정상적으로 읽혀야 한다.
- 주요 화면에서 `undefined`, 레이아웃 깨짐, 과도한 가로 스크롤이 없어야 한다.
- 공통 상태 표현과 스타일 규칙이 문서와 코드에 일관되게 반영되어야 한다.
