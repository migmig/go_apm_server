# 개선점 분석 (2026-03)

## 1) 즉시 개선 권장 (P0)

### 1.1 WebSocket 보안 옵션 제거 (`InsecureSkipVerify: true`)
- 현재 WebSocket 수락 옵션에 `InsecureSkipVerify: true`가 설정되어 있어, 기본 정책으로는 불필요하게 완화된 설정이 고정되어 있습니다.
- `nhooyr/websocket`의 `AcceptOptions`는 기본값으로도 동작하므로, 특별한 사유가 없다면 옵션 자체를 제거하는 편이 안전합니다.
- 개선안
  - `HandleWebSocket`에서 `AcceptOptions`를 최소화(또는 nil)하고, 필요한 origin 정책만 명시적으로 허용.
  - 운영환경/개발환경을 분리해 허용 origin을 설정값으로 제어.

### 1.2 API 테스트/빌드 결합도 완화 (`web/dist` 의존성)
- `go test ./...` 실행 시 `web/embed.go`의 `//go:embed all:dist`로 인해 `dist` 폴더가 없으면 테스트가 실패합니다.
- 프론트엔드 산출물이 없는 CI 단계에서 백엔드 테스트만 빠르게 돌리기 어렵습니다.
- 개선안
  - 빌드 태그(`embed_ui`)를 도입해, UI 임베드 파일과 스텁 파일을 분리.
  - 기본 테스트 경로에서는 스텁을 사용하고, 릴리즈 빌드에서만 실제 임베드 사용.

## 2) 단기 개선 권장 (P1)

### 2.1 `/api/system`의 디렉터리 용량 계산 캐시화
- `HandleGetSystem` 호출 시마다 `calcDirSize`가 전체 디렉터리를 순회합니다.
- 데이터 디렉터리가 커질수록 API 응답 지연 및 I/O 부하가 커질 수 있습니다.
- 개선안
  - 최근 계산값을 메모리에 캐시하고(예: 30~60초 TTL), 주기적 백그라운드 갱신으로 전환.
  - 실패 시 직전 성공값을 fallback으로 사용.

### 2.2 API 버전 문자열 하드코딩 제거
- `/api/system` 응답의 버전이 코드에 문자열 상수(`v0.7.0-alpha`)로 하드코딩되어 있습니다.
- 릴리즈 자동화 시 버전 누락/불일치 위험이 있습니다.
- 개선안
  - `-ldflags "-X main.version=..."` 방식으로 주입하거나, `internal/version` 패키지로 중앙화.

### 2.3 쿼리 파라미터 검증 강화 (limit/offset)
- `limit`, `offset`은 파싱 실패 시 기본값을 쓰지만, 상한/하한 클램프가 보이지 않습니다.
- 비정상적으로 큰 요청이 들어오면 스토리지 조회 비용이 급증할 수 있습니다.
- 개선안
  - endpoint 공통으로 `limit` 상한(예: 200~1000) 적용.
  - 음수 값은 0 또는 기본값으로 강제.

## 3) 중기 개선 권장 (P2)

### 3.1 API 계층 테스트 독립성 강화
- `internal/api` 테스트가 웹 임베드 조건에 영향을 받아 실행이 막히는 구조입니다.
- 개선안
  - 정적 파일 서빙/임베드 초기화를 별도 인터페이스로 추출.
  - 핸들러 유닛 테스트에서는 스텁 핸들러를 주입해 API만 검증.

### 3.2 관측성 강화: 핵심 경로 메트릭 추가
- 이미 `/metrics`는 제공되지만, API별 지연/오류율 및 큐 적체 지표를 더 촘촘히 노출하면 병목 분석이 쉬워집니다.
- 개선안
  - API 라우트별 latency histogram, status code counter 추가.
  - batch queue depth / dropped count를 라벨 최소화 원칙으로 노출.

## 우선순위 실행 순서 제안
1. **P0-1** WebSocket 옵션 정리 (보안/정책 명확화)

3. **P1-1** `/api/system` 용량 계산 캐시화 (성능)
4. **P1-2, P1-3** 버전/파라미터 검증 정리 (운영 안정성)
