# Phase 4 리서치 및 사용자 사용성(UX) 개선 기획안

Phase 1~3의 структур적/시각적 컴포넌트 개선 작업 이후, APM(Application Performance Monitoring) 도구로서 실사용 시 체감할 수 있는 사용성을 극대화하기 위한 추가 제안 사항입니다.

## 1. 개선 동기 및 목표
현재 대시보드는 실시간 데이터를 안정적으로 띄우고, 가상화(Virtualization) 기술을 통해 다량의 데이터를 표출하는 수준에 도달했습니다. 그러나 실제로 장애를 추적(Troubleshooting)하는 상황의 디버깅 효율을 높이는 과정에서 필수적인 **텍스트 복사**, **키워드 스캔 능력**, **데이터 정렬 및 화면 동결**과 같은 핵심 도구(Micro-interaction)들이 부재합니다. 본 문서는 이를 해결하기 위한 5가지 개선 과제를 제안합니다.

---

## 2. 구체적인 개선 과제 목록

### 2.1. 원클릭 클립보드 복사 (Copy-to-Clipboard)
장애 추적 시 Trace ID를 다른 시스템(Splunk, Kibana, Jira 이슈 티켓 등)으로 전달하기 위해서는 ID 복사가 필수적입니다.
- **As-Is**: 단순 텍스트 랜더링만 제공되어 마우스로 텍스트 시작과 끝을 정교하게 드래그해야 복사 가능 (폭 좁은 화면에서 오류 잦음).
- **To-Be**:
  - `Trace ID`, `Span ID`, `요청 시간` 및 `LogAttributes` 태그에 **복사 아이콘(Copy Icon)**을 노출시킵니다.
  - 클릭 시 `navigator.clipboard.writeText`를 수행하고, Phase 3에서 도입된 `toast.success('복사되었습니다')`를 통해 시각적 만족감을 제공합니다.
- **예상 난이도**: 하 (단순 컴포넌트 추가 및 브라우저 API 호출 변경)

### 2.2. 로그 검색어 실시간 하이라이팅 (Keyword Highlighting)
수백 줄의 로그 더미 속에서 특정 오류 키워드(예: `connection timeout`)를 입력하여 필터링 한 뒤에도, 정작 어느 부분에서 발생했는지 육안으로 훑어야 하는 불편함이 있습니다.
- **As-Is**: 데이터 렌더링 시 일치하는 텍스트 문자열에 대한 추가적인 시각 처리가 없음.
- **To-Be**:
  - 문자열 변환 컴포넌트(`<HighlightText />`)를 신규 생성합니다.
  - 검색어 상태(`searchBody`)와 일치하는 부분만 `<span>`으로 분할하여 형광펜 처리(`bg-yellow-500/30 text-yellow-200`)합니다. 대소문자를 구분하지 않는 정규식 분할 로직을 적용합니다.
- **예상 난이도**: 중 (정규식 기반 텍스트 분할 알고리즘 구현 필요)

### 2.3. 트레이스 목록 기준별 테이블 정렬 (Table Sorting)
단순한 목록 확인을 넘어, 가장 오래 걸린 요청(지연성 병목 지점)부터 파악하는 것은 APM의 First Step입니다.
- **As-Is**: `Traces.tsx` 페이지에서 단순 최신순 데이터 배열(`start_time` DESC)만 노출합니다.
- **To-Be**:
  - 테이블 헤더(수행 작업, 소요 시간, 결과 등)에 오름차순/내림차순 토글 화살표를 배치합니다.
  - `Traces.tsx` 내의 메모리 배열 정렬 혹은 상단 API Parameter를 통해 `sort` 방식을 전환합니다. 초기엔 프론트엔드 메모리 소팅만 지원해도 유용합니다 (`trace.duration_ms`).
- **예상 난이도**: 하 (프론트엔드 JS `.sort()` 로직 추가 및 화살표 UI)

### 2.4. 수신 화면 동결 토글 (Global Freeze / Pause)
쏟아지는 로그나 트레이스 데이터에 의해 현재 관찰 중이던 영역이 밀려나거나 선택된 요소를 상실하는 것을 막아야 합니다.
- **As-Is**: 웹소켓 메시지가 올 때마다 상단에 데이터가 밀리며 렌더링 됩니다. (Logs 화면 일부에만 구현)
- **To-Be**:
  - 글로벌 화면 좌측 네비게이션바 혹은 각 페이지 상단 헤더 영역에 `실시간 수신 켜기/끄기(Live / Paused)` 버튼을 노출합니다.
  - 끄기 상태에선 웹소켓 연결은 유지하여 데이터를 버퍼에 모아두되 화면에는 반영하지 않습니다. 켜기 상태로 복원 시 버퍼 데이터가 일괄 렌더링 됩니다.
- **예상 난이도**: 상 (WebSocket 커스텀 훅 `useWSMessage` 내부에 상태 버퍼링 관리 로직 주입 필요)

### 2.5. 워터폴 차트 미니맵 및 Zoom 드래그
대규모 MSA 환경에서 100개 이상의 스팬이 찍힐 경우, 전체 수행 시간 대비 비정상적으로 긴 병목 구간을 빠르게 파악하기 어렵습니다.
- **As-Is**: 전체 Timeline 대비 Span `width`와 `left`만 구해서 렌더링. 구간 확장이 불가능합니다.
- **To-Be**:
  - 워터폴 리스트 상단에 작은 형태의 미니맵(Overview Timeline)을 추가 표시합니다.
  - 향후 D3.js나 스크롤 제어를 활용해 돋보기처럼 드래그한 영역 시간대만 확대하는 기능 추가 기획 (이 과제는 중장기 백로그로 분류).
- **예상 난이도**: 매우 상 (마우스 드래그 이벤트, 차트 스케일링 동적 변경)

---

## 3. 진행 방법 제안
가시적인 효과가 높은 **2.1. 아이디/속성 복사 (Copy-to-Clipboard)** 와 **2.2. 로그 하이라이팅**, 그리고 사용자가 자주 묻는 **2.3. 테이블 정렬(특히 '소요 시간' 기준)** 을 Phase 4 작업으로 먼저 묶어 구현하는 것을 권장합니다. 동결 토글이나 미니맵은 Phase 5로 이관하여 별도의 컴포넌트 레벨업 설계가 필요합니다.
