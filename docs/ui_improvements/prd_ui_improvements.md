# APM Server Web UI 개선사항 PRD (Product Requirements Document)

## 1. 개요
현재 구현된 React 기반 APM 대시보드의 유지보수성, 성능, 그리고 사용자 경험(UX)을 끌어올리기 위한 UI/UX 개선 프로젝트입니다. 본 문서는 컴포넌트 구조 최적화부터 렌더링 성능 이슈 해소까지 단기 및 중장기적으로 개선해야 할 요구사항을 정의합니다.

## 2. 목표
- **코드 재사용성 향상**: 중복된 컴포넌트와 비즈니스 로직을 하나로 통합하여 유지보수성 증대
- **데이터 표현의 정확성 확보**: 하드코딩된 계산식을 제거하고, 데이터 시각화의 정확도 개선
- **렌더링 성능 최적화**: 대량의 로그 및 트레이스 데이터를 효율적으로 렌더링하도록 DOM 로드 최소화
- **UX 및 접근성 개선**: 모바일 접근성, 예외 상황 처리(에러 알림 등) 등 사용자 편의성 제고

## 3. 세부 요구사항 (Requirements)

### 3.1. 컴포넌트 및 로직 공통화 (Consistency & Reusability)
- **통계 카드(Stat Card) 분리**
  - **As-Is**: `Dashboard.tsx`와 `ServiceDetail.tsx`에서 통계 카드를 각각 별도로 구현하여 사용
  - **To-Be**: `components/ui/StatCard.tsx`와 같은 공통 컴포넌트로 분리하여 동일한 디자인과 로직 유지
- **서비스 상태(Health) 판단 로직 공통화**
  - **As-Is**: 오차율(Error Rate > 0.05) 및 응답 지연(Latency > 500ms) 등의 위급 상태 판단 로직 코드가 중복됨
  - **To-Be**: `lib/health.ts` 등의 유틸리티 함수로 추출하여 전역적으로 통일된 기준을 사용하도록 보장
- **상태 및 심각도 색상 토큰화**
  - **As-Is**: 성공/실패, 로그 심각도(Error, Warn 등)에 따른 Tailwind 디자인 클래스가 분산되어 하드코딩됨
  - **To-Be**: 디자인 토큰 형태의 공통 함수 및 상수로 분류하여 일관된 색상 테마 제공

### 3.2. 대시보드 데이터 표현 개선 (Dashboard Presentation)
- **동적 RPS 계산 방식 도입**
  - **As-Is**: RPS가 초 단위가 아닌 `(span_count / 3600)`으로 고정되어 계산표시 중
  - **To-Be**: 선택된 시간 범위(Time Window)에 동적으로 대응하여 올바른 RPS 값을 계산하거나 백엔드 메타데이터 기반으로 연산
- **스파크라인 차트 데이터 매핑 수정**
  - **As-Is**: `Dashboard`의 4가지 주요 지표 스파크라인이 모두 같은 데이터키(`rps`)를 재사용함
  - **To-Be**: '모든 요청 수', 'Span 수', '평균 응답 시간' 등 각각의 지표에 알맞은 시계열 데이터를 속성 매핑

### 3.3. 트레이스 상세 뷰 (Waterfall Chart) 성능 및 시각적 개선
- **반응형 컨테이너 레이아웃 수정**
  - **As-Is**: 컨테이너의 제한된 너비 분할 방식으로 인해 텍스트 잘림 현상 발생 (특히 좁은 화면)
  - **To-Be**: 좌측 라벨 영역을 고정 너비(예: `w-64`)로 설정하여 식별 영역을 보장하고, 우측 타임라인 영역만 오버플로우 스크롤되도록 개선
- **대용량 데이터 렌더링 처리 및 인터랙션**
  - **As-Is**: 스팬(Span) 개수가 많은 워터폴 차트에서 렌더링 병목 및 직관적 식별의 한계
  - **To-Be**: 워터폴 차트의 활성화(Active)된 스팬에 대해 명확한 배경 강조 가이드를 부여하고, 장기적으로 줌(Zoom) 기능 등 도입 고려

### 3.4. 로그 뷰어 최적화 (Logs Viewer)
- **리스트 가상화(Virtualization) 도입**
  - **As-Is**: 실시간 배열 내 최대 500개 로그 렌더링 시 DOM 요소 폭증으로 인한 메모리/버벅임 병목 우려
  - **To-Be**: 화면에 보이는 요소만 렌더링하는 가상화 스크롤 라이브러리(`react-virtualized` 또는 `react-virtuoso`) 적용
- **속성(Attributes) 태그 오버플로우 제어**
  - **As-Is**: 다수의 Key-Value 태그가 나열될 때 화면을 과도하게 차지해 본문 가독성 저하
  - **To-Be**: 일정 개수 이상의 태그는 줄임표 단축 또는 [더보기/접기] 버튼을 통해 확장 가능하도록 변경

### 3.5. UX 및 접근성 최적화
- **검색 폼 엔터키 이벤트 접근성 개선**
  - **As-Is**: UI의 검색창 로직이 `<input onKeyDown={...}>`을 사용하여 제어됨
  - **To-Be**: `<form onSubmit={...}>` 구조로 래핑하여 모바일 가상 키보드의 '검색' 및 다양한 접근성 기술 호환성 개선
- **비간섭형 에러 알림(Toast Notification) 도입**
  - **As-Is**: 웹소켓 폴링 중 일시적 API 실패 시 상단 전체 공간을 밀어내며 `StatusBanner` 노출 (작업 가시성 방해)
  - **To-Be**: 글로벌 Toast 메시지 UI를 구현하고, 일시적인 에러 처리는 화면 흐름을 방해하지 않는 Toast 형태로 변경

## 4. 마일스톤 및 릴리스 전략
- **Phase 1 (Quick Wins)**: 공통화 작업(StatCard, 상태 로직 분리 및 색상 토큰화), 폼 접근성, 스파크라인 지표 등 즉각적 개선사항 처리
- **Phase 2 (Performance)**: 로그 뷰어 UI Virtualization 시스템 적용, 워터폴 차트 레이아웃 CSS 및 스크롤 개선
- **Phase 3 (UX Enhancement)**: Toast Notification 도구 적용, 태그 접기/펼치기 등 사용자 조작 편의성 추가
