# APM Server UI 사용성(Usability) 심층 분석

## 1) 분석 목적과 범위
본 문서는 현재 Web UI의 핵심 사용자 흐름(모니터링 진입 → 이상 징후 인지 → 원인 탐색 → 트레이스/로그 상세 확인)에서 발생하는 사용성 저하 요인을 정리하고, **업무효율 관점의 우선순위 개선안**을 제시한다.

- 대상 화면: Dashboard, Service Detail, Trace Detail(Waterfall), Logs Viewer, Search 입력 UX
- 대상 사용자: 온콜 엔지니어, 백엔드 개발자, SRE
- 분석 관점: 학습 용이성(Learnability), 탐색 효율(Efficiency), 오류 예방(Error Prevention), 인지 부하(Cognitive Load), 접근성(Accessibility)

---

## 2) 핵심 사용자 여정(critical journey) 기준 평가

### Journey A: “지금 장애인가?”를 10초 내 판단
1. 사용자는 대시보드에서 서비스 건강도와 추세를 빠르게 훑는다.
2. 이상 신호(에러율 증가, 지연 증가)를 발견하면 해당 서비스로 이동한다.

**현재 리스크**
- 지표별 시계열 의미가 불명확하면(스파크라인 매핑 오류) 신뢰도가 낮아져 판단이 지연됨.
- RPS 계산이 시간 범위에 비례하지 않으면 체감 트래픽과 숫자가 어긋나 오탐/미탐 유발.

**개선 효과**
- 동일 시간창 기준으로 지표 의미를 맞추면 “이상 여부 인지 시간” 단축.
- 지표 정의 일관화로 팀 내 커뮤니케이션 비용 감소.

### Journey B: “어디서 느린가?”를 1분 내 파악
1. Service Detail에서 병목 가능성이 높은 Trace로 진입.
2. Waterfall에서 긴 Span과 병렬 흐름을 비교해 병목 구간을 찾음.

**현재 리스크**
- 좁은 화면에서 레이블이 잘리면 span 식별 실패 → 클릭/비교 횟수 증가.
- 대량 span 렌더링 시 스크롤/하이라이트 반응성이 낮아 분석 몰입도가 떨어짐.

**개선 효과**
- 고정 레이블 + 스크롤 분리로 식별 비용 절감.
- 활성 span 강조/집중 기능으로 병목 탐지 정확도 상승.

### Journey C: “왜 실패했는가?”를 로그로 확인
1. Logs Viewer에서 에러 레벨을 필터링.
2. 개별 로그의 attributes를 확인해 root cause 후보를 수집.

**현재 리스크**
- 태그가 과다 노출되면 한 행이 비정상적으로 커져 시선 스캔이 어려움.
- 대량 로그 DOM 렌더링으로 스크롤 끊김 발생 시 탐색 효율 급감.

**개선 효과**
- 가상화 + 태그 접기/펼치기 도입 시 “정보량 대비 가독성” 균형 회복.
- 로그 탐색의 연속성이 좋아져 원인 파악 리드타임 단축.

## 3) 사용성 문제를 휴리스틱 관점으로 재정리

### A. 시스템 상태의 가시성 (Visibility of system status)
- 문제: 일시적 에러가 상단 배너로 레이아웃을 밀어내면 사용자의 현재 집중 맥락이 깨짐.
- 개선: 비간섭형 Toast + 재시도 상태 표시(예: “재연결 중…”)로 정보는 제공하되 작업 흐름은 유지.

### B. 현실 세계와의 일치 (Match with real world)
- 문제: RPS/스파크라인 의미 불일치는 사용자의 운영 감각(실제 트래픽 변화)과 UI 숫자의 괴리를 만듦.
- 개선: 시간창 종속 계산 + 툴팁에 정의(분모/집계 방식) 명시.

### C. 사용자 제어와 자유 (User control & freedom)
- 문제: 속성 태그가 항상 전체 노출이면 사용자는 “요약 보기”를 선택할 수 없음.
- 개선: 기본 요약(상위 N개) + 펼치기 제어 제공.

### D. 일관성과 표준 (Consistency & standards)
- 문제: 건강도 판단 로직/색상 체계가 화면마다 다르면 동일 상태를 다르게 해석할 수 있음.
- 개선: 상태 로직과 컬러 토큰을 공통화해 인지 규칙을 고정.

### E. 오류 예방 (Error prevention)
- 문제: 입력창의 Enter 처리 방식이 플랫폼마다 다르게 동작하면 검색 실행 실패 가능.
- 개선: form submit 기반으로 표준 동작 보장.

### F. 인식 기반 사용 (Recognition rather than recall)
- 문제: 워터폴에서 레이블 인지가 어렵거나 하이라이트가 약하면 사용자가 span 맥락을 기억에 의존.
- 개선: 고정 식별영역, 선택 상태 시각 강조, hover/focus 피드백 강화.

## 4) 우선순위 제안 (사용성 임팩트 × 구현 난이도)

### P0 (즉시)
1. **RPS/스파크라인 의미 정합성 확보**
2. **검색 입력의 form submit 전환**
3. **상태/심각도 컬러 및 health 로직 공통화**

> 기대 효과: 빠른 신뢰도 회복 + 초기에 체감되는 사용성 개선.

### P1 (단기)
1. **Waterfall 레이아웃 분리(고정 레이블/가변 타임라인)**
2. **Logs virtualization 도입**
3. **Attributes 접기/펼치기**

> 기대 효과: 관찰/탐색 속도 향상, 대량 데이터 상황에서 피로도 감소.

### P2 (중기)
1. **비간섭형 Toast + 에러 상태 분류(치명/일시)**
2. **Trace 분석 보조 기능(줌, 범위 선택, mini-map) 검토**

> 기대 효과: 고난도 장애 분석 시 작업 연속성 강화.

## 5) 측정 가능한 사용성 KPI 제안
개선의 성과는 “느낌”이 아니라 수치로 추적해야 한다.

- **TTFI (Time To First Insight)**: 대시보드 진입 후 이상 징후를 처음 인지하기까지 시간
- **TTFP (Time To Find bottleneck Point)**: Trace 진입 후 병목 span 특정까지 시간
- **Log Scan Throughput**: 분당 확인 가능한 로그 행 수
- **Interaction Error Rate**: 검색 실행 실패, 잘못된 탭 이동, 재클릭 비율
- **UI Jank Ratio**: 스크롤 중 프레임 드랍/지연 이벤트 비율

권장 방식: 개선 전/후 동일 시나리오의 태스크 기반 사용성 테스트(5~8명)와 프론트엔드 성능 지표를 함께 측정.

## 6) 실행 체크리스트 (팀 운영 관점)
- [ ] 지표 정의서(메트릭 명칭/분모/집계 주기)를 UI 툴팁/문서와 일치시켰는가?
- [ ] 건강도 색상과 임계치가 전 화면에서 동일하게 동작하는가?
- [ ] 로그/워터폴 대량 데이터에서 스크롤 FPS 저하가 허용 기준 이내인가?
- [ ] 키보드 중심 사용(Enter, Tab, Focus ring)에서 막힘이 없는가?
- [ ] 경고/오류 알림이 작업 맥락을 방해하지 않는가?

---

## 7) 결론
현재 개선 백로그는 방향이 적절하며, 특히 **데이터 의미 정합성(신뢰성) + 대량 데이터 탐색 성능 + 비간섭형 피드백**이 사용성의 핵심 축이다. 우선순위를 P0→P1→P2로 실행하면, 사용자 입장에서 가장 체감이 큰 “판단 속도”와 “분석 정확도”를 빠르게 끌어올릴 수 있다.
